Analysis of Current Filename Parsing Logic and Improvement Recommendations

CURRENT LIMITATIONS IN extract_metadata_from_filename():
1. Only recognizes " - " as artist-title separator
2. Doesn't handle common alternative separators like " – " (en dash), " — " (em dash), or "_"
3. Doesn't handle common patterns like:
   - "[handle] Track.wav" (SoundCloud style)
   - "Title [handle].wav" (SoundCloud style)
   - "Artist_Title.wav" (underscore separator)
   - "Artist - Title (Remix).wav" (already handled reasonably well)
   - "Tracknumber - Artist - Title.wav" (common in albums)
   - "Artist - Title feat. Guest.wav" (already handled)

RECOMMENDED IMPROVEMENTS:

1. ENHANCE extract_metadata_from_filename() to recognize multiple separators:
   - " - " (hyphen with spaces) - current
   - " – " (en dash with spaces)
   - " — " (em dash with spaces)
   - "_" (underscore)
   - "." (dot, when not file extension)

2. ADD SPECIAL PATTERN RECOGNITION:
   - "[handle] Title.wav" → Artist: (from handle lookup), Title: Title
   - "Title [handle].wav" → Artist: (from handle lookup), Title: Title
   - "TrackNN - Artist - Title.wav" → Skip track number, Artist: Artist, Title: Title
   - Handle "Various Artists" compilations

3. IMPROVE HANDLE EXTRACTION LOGIC:
   Current HANDLE_RE = re.compile(r'\[([a-z0-9_]+)\]', re.IGNORECASE)
   Limitations:
   - Doesn't allow dashes in handles (common in SC: [user-name])
   - Doesn't allow dots in handles
   - Could be expanded to: r'\[([^\]]+)\]' to capture anything between brackets

4. ENHANCE CLEAN_TITLE_FOR_SEARCH():
   Current implementation is complex but functional. Could simplify by:
   - Using a more straightforward regex approach
   - Better handling of nested brackets
   - Preserving meaningful parts while removing ONLY remix/edit/etc. indicators

SPECIFIC CODE CHANGES NEEDED:

In extract_metadata_from_filename():
- Add multiple separator recognition
- Add special pattern detection for handle-leading and handle-trailing filenames
- Add track number skipping logic

In HANDLE_RE:
- Expand to allow more characters in handles: r'\[([^\]]+)\]'

These changes would make the converter much more robust against real-world filename variations commonly found in music collections.
